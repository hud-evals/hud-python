An environment is everything an agent can interact with—your APIs, services, databases, wrapped as tools—plus the logic that defines how agents are *evaluated*. When you deploy an environment, you're creating an isolated sandbox that agents can learn from at scale.

## Why Environments?

Your production API is a single live instance with shared state—you can't run 500 tests against it in parallel without causing chaos. Environments spin up fresh for every evaluation: isolated, deterministic, reproducible. Run thousands in parallel, each starting from the exact state you define, each generating training data.

## Core Concepts

An environment has three building blocks:

### Tools

Tools are functions that agents can call. Any Python function becomes a tool with `@env.tool()`:

```python
from hud import Environment

env = Environment("my-env")

@env.tool()
async def search(query: str) -> str:
    """Search the knowledge base."""
    return db.search(query)
```

Tools are how agents interact with your system. They see the function signature, the docstring, and the return type—just like a developer reads an API reference. Design tools that are self-explanatory and return actionable data.

<Tip>
Before building custom tools, check if [HUD's pre-built tools](/tools) already provide what you need—computer control, shell execution, file editing, web browsing, and more.
</Tip>

### Scenarios

Scenarios define *what to ask* and *how to score*. Two `yield` statements capture both:

```python
@env.scenario("checkout")
async def checkout_flow(product_name: str):
    answer = yield f"Add '{product_name}' to cart and complete checkout"
    
    order_exists = await check_order_status(product_name)
    yield 1.0 if order_exists else 0.0
```

The agent runs between the yields. First yield sends the prompt and returns the agent's answer. Second yield checks environment state and returns a reward. Scenario parameters become the task arguments—create many tasks from one scenario by varying the inputs.

### Connections

You don't have to build everything from scratch. Connect existing infrastructure and it becomes tools automatically:

```python
env.connect_fastapi(app)                                    # FastAPI → tools
env.connect_openapi("https://api.example.com/openapi.json") # OpenAPI spec → tools
env.connect_hub("hud-evals/browser")                        # HUD Hub environments
env.connect_image("my-service:v1")                          # Docker images
```

## Getting Started

Scaffold a new environment with `hud init`:

```bash
hud init
```

This creates an `env.py` with a sample tool and scenario you can run immediately. The golden path from there:

1. **Edit `env.py`** — add your tools and scenarios
2. **`hud dev`** — run as an MCP server, connect from Cursor or Claude Code
3. **`hud eval`** — create tasks and run evaluations
4. **`hud deploy`** — deploy to the platform for scale

### hud dev

Spawn your environment as an MCP server that any MCP client can connect to:

```bash
hud dev env:env
```

The `env:env` syntax is like uvicorn—`module:attribute`. It tells `hud dev` to import `env.py` and run the `env` object as an MCP server. Add `-w` to watch files for hot-reload:

```bash
hud dev env:env -w env.py -w tools/
```

Then in Cursor's MCP settings:

```json
{
  "my-dev-env": { "url": "http://localhost:8765/mcp" }
}
```

### Running an Agent Programmatically

Run a full agent loop locally—this mirrors exactly what happens in remote rollouts:

```python
result = await env("count", sentence="Strawberry", letter="r").run("claude-sonnet-4-5")
print(f"Reward: {result.reward}")
```

Or build your own agent loop with format converters. See [Integrations](/guides/integrations) for OpenAI, Anthropic, LangChain, and more.
