HUD environments aren't limited to single-turn evals. You can build multi-turn chat experiences where users talk to your environment over multiple turns, and the environment owns the conversation flow.

This guide walks through the full setup: defining a chat scenario, serving it over A2A, and connecting a client.

## Defining a Chat Scenario

A chat scenario is a regular `@env.scenario()` with `chat=True`. This tells HUD to pass prior conversation history into the scenario on every turn.

```python
from typing import Any
from hud import Environment

env = Environment("support")

@env.scenario("help", chat=True)
async def help_chat(messages: list[dict[str, Any]] | None = None):
    messages = messages or []
    yield f"You said: {messages[-1]['content'] if messages else 'nothing yet'}"
    yield 1.0
```

Key points:

- **`chat=True`** — HUD enforces that the scenario accepts a `messages` parameter.
- **`messages`** — Prior turns are passed in automatically. The scenario can read history to maintain context.
- **The environment owns the flow** — The scenario decides what to ask, when to ask it, and how to use the tools. No SDK-side routing needed.

## Serving Over A2A

Once you have a chat scenario, you can serve it as an [A2A (Agent-to-Agent)](https://google.github.io/A2A/) endpoint using `ChatService`. This is a thin transport layer that:

- Receives user messages from A2A clients
- Maps each conversation to a `Chat` session
- Forwards every turn to your configured scenario
- Returns the response

```python
from hud.services import ChatService

service = ChatService(
    "support",
    model="claude-haiku-4-5",
    scenario="help",
)
service.serve(host="0.0.0.0", port=9999)
```

Or with environment variables for flexibility:

```bash
HUD_ENV=support HUD_SCENARIO=help \
    uv run python examples/03_a2a_chat_server.py
```

The chat service publishes an agent card at `/.well-known/agent-card.json` and accepts A2A messages at the root endpoint.

<Note>
Each `ChatService` targets exactly one scenario. If your environment has multiple chat-compatible scenarios, run one chat service per scenario or build client-side routing to pick which one to call.
</Note>

## Using Chat Directly (No A2A)

If you don't need the A2A protocol, you can use `Chat` directly in Python:

```python
from hud.services import Chat
from hud.eval.task import Task

chat = Chat(
    Task(env={"name": "support"}, scenario="support:help"),
    model="claude-sonnet-4-20250514",
)

r1 = await chat.send("Look into account ABC-123")
print(r1.content)

r2 = await chat.send("What's their current plan?")
print(r2.content)
```

`Chat` manages history internally. Each `send()` creates a fresh eval context, passes accumulated history to the scenario, runs the agent, and returns the trace.

## Connecting a Client

Any A2A-compatible client can talk to your chat service. See [`examples/04_a2a_chat_llm_client.py`](https://github.com/hud-evals/hud-python/blob/main/examples/04_a2a_chat_llm_client.py) for a full example that puts an LLM in front and calls the service as a tool.

The pattern is: resolve the agent card, create a client, and stream messages back.

## When to Use What

| Approach | When |
|----------|------|
| **`Chat` directly** | Python scripts, notebooks, tests — no network layer needed |
| **`ChatService`** | Serve over HTTP for web UIs, other agents, or cross-language clients |
| **LLM + A2A client** | When you want a conversational AI layer in front that decides when to call the environment |
